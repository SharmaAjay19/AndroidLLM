# AndroidLLM — Feature Build Plan (2026‑01‑07)

Plan for four features that turn AndroidLLM into a genuine productivity/utility app.
Each builds on what already exists: on‑device Qwen3‑4B (llama.cpp), the JSON agent
tool‑loop, web tools (`web_search`/`fetch_url`), file tools + workspace, Room chat
storage, and Whisper voice input.

**Priority order:** Share‑to‑assistant → Phone‑native tools → Scheduled prompts → On‑device RAG
(smallest‑effort/highest‑utility first; RAG last as it needs a new model + vector store).

---

## Feature 1 — Share‑to‑assistant (system‑wide entry)

**Goal:** Use the assistant from *any* app — share/select text, a URL, or a file and
run "Summarize / Translate / Extract action items / Answer".

**UX**
- Appears in the Android share sheet ("Ask AndroidLLM") and in the text‑selection
  toolbar ("Process text").
- A lightweight entry screen: shows the shared content + quick‑action chips
  (Summarize, Key points, Translate, Reply draft, Ask…) → opens a new chat with the
  result streaming.

**Architecture / components**
- New `ShareActivity` (or reuse `MainActivity` with intent routing).
- Manifest intent filters: `ACTION_SEND` (`text/plain`, `*/*`), `ACTION_SEND_MULTIPLE`,
  and `ACTION_PROCESS_TEXT`.
- Intent handling maps content → existing engine:
  - Shared **text** → seed a new chat as the user message (+ chosen action prefix).
  - Shared **URL** → call existing `fetch_url`, then summarize.
  - Shared **file** → copy into the workspace, then `read_file` (reuse chunking).

**Implementation steps**
1. Add intent filters + a `ShareEntry` composable/activity.
2. Parse `Intent.EXTRA_TEXT` / `EXTRA_STREAM` / `EXTRA_PROCESS_TEXT`.
3. Reuse `MainViewModel.send()` with a preset action prompt; ensure the model is loaded
   (prompt to download if not).
4. Quick‑action chips → prompt templates.

**Permissions:** none new (URIs granted via the share intent).

**Testing**
- Instrumented: fire `ACTION_SEND` with sample text/URL and assert a chat is created
  with the expected seeded content (no LLM assertion needed for routing).
- Manual: share from Chrome (URL) and a file manager (file).

**Effort:** S (1–2 days). **Risk:** Low. Mostly manifest + intent plumbing over existing engine.

## Feature 2 — Phone‑native agent tools

**Goal:** Let the LLM act on the phone: reminders, calendar, contacts, clipboard,
and draft SMS/email — via the existing tool‑calling loop.

**Tools to add** (JSON, same mechanism as `read_file`/`web_search`)
- `create_reminder {text, time}` — `AlarmManager` + a notification.
- `read_calendar {date_range}` / `create_event {title, start, end}` — `CalendarContract`.
- `find_contact {name}` — `ContactsContract` (read).
- `read_clipboard {}` — `ClipboardManager`.
- `compose_message {to, body, channel}` — **draft‑only** via `ACTION_SENDTO`/intent
  (user confirms send; never auto‑send).

**Architecture / components**
- Extend `Tools`/`Agent.kt`: register names, labels, and a `PhoneTools` executor.
- Route these (like web tools) in `MainViewModel` since some need a `Context`/main thread.
- Per‑tool runtime permission, requested on first use (READ_CALENDAR, WRITE_CALENDAR,
  READ_CONTACTS, POST_NOTIFICATIONS, SCHEDULE_EXACT_ALARM).
- A **confirmation step** for state‑changing tools (create event/reminder/message):
  show a confirm bubble before executing — safety + trust.

**Implementation steps**
1. Add tool schemas to the system prompt (behind the Tools toggle).
2. Implement each provider (Calendar/Contacts/Alarm/Clipboard/Intent).
3. Permission gating + graceful "permission needed" tool results.
4. Confirmation UI for writes.

**Testing**
- Unit: argument parsing + safe path handling (host tests, like `WorkspaceResolveTest`).
- Instrumented: `create_reminder`/`create_event` against the provider, then read back and
  assert; `read_clipboard` set‑then‑read. Deterministic, no LLM needed.

**Effort:** M (3–5 days). **Risk:** Medium (permissions, OEM quirks). Draft‑only + confirms
mitigate safety risk.

## Feature 3 — Scheduled prompts (proactive briefings)

**Goal:** Run a saved prompt on a schedule (e.g., a daily 8am briefing that combines
`web_search` + calendar) and deliver the result as a notification.

**UX**
- "Schedules" screen: create a schedule = { name, prompt, cadence (daily/weekly/cron‑lite),
  time, deliver‑as: notification/new chat }.
- Tapping the notification opens the generated chat.

**Architecture / components**
- **WorkManager** periodic work (or `AlarmManager` for exact‑time) → a worker that:
  loads the model, runs the prompt through the agent loop (tools enabled), writes a chat,
  posts a notification.
- Persist schedules in Room (new `schedules` table).
- Foreground service / expedited work while the model runs (LLM inference is heavy).

**Implementation steps**
1. `schedules` entity + DAO + simple UI.
2. `BriefingWorker` (WorkManager) that reuses `MainViewModel`'s generation path
   (refactor the agent loop into a reusable engine callable off‑UI).
3. Notification channel + tap‑to‑open deep link.
4. Battery/thermal guards: skip if low battery; coalesce.

**Implementation note:** requires extracting the agent/generation loop out of the
ViewModel into a headless `ChatEngine` so both UI and the worker can call it.

**Testing**
- Unit: schedule CRUD + next‑run computation.
- Instrumented: enqueue a one‑shot worker with a trivial prompt (tools off) and assert a
  chat row + notification are produced.

**Effort:** M (3–5 days, +1 for the ChatEngine refactor). **Risk:** Medium (background
execution limits, running a big model in a worker).

## Feature 4 — Chat with your documents (on‑device RAG)

**Goal:** Ask questions across your own files/PDFs/screenshots, fully offline, by
retrieving relevant snippets before answering.

**Architecture / components**
- **Embeddings model**: small GGUF embedder (e.g., `bge‑small`/`e5‑small`) via the ggml
  stack already used — likely a new isolated native module like `:whisper`, or llama.cpp's
  embedding API.
- **Indexer**: chunk workspace files (reuse `read_file` chunking) → embed → store vectors.
- **Vector store**: SQLite (already present) with cosine search (brute force is fine for
  personal scale; add ANN later if needed).
- **OCR**: ML Kit Text Recognition for images/screenshots/scanned PDFs → text → index.
- **Tool**: `search_documents {query, k}` returns top‑k snippets; the agent calls it, then
  answers grounded in the results (keeps within the 2K context via retrieval).

**Implementation steps**
1. Add the embedding engine + a `Embedder` API.
2. Indexing pipeline (on file add/upload) + a re‑index action.
3. Vector table + similarity search.
4. `search_documents` tool wired into the agent loop.
5. ML Kit OCR path for image/PDF inputs.

**Testing**
- Unit: chunking + cosine ranking on synthetic vectors (host).
- Instrumented: index a bundled text file, query a known phrase, assert the right snippet
  ranks first; OCR a bundled image and assert extracted text.

**Effort:** L (1–2 weeks). **Risk:** Medium‑High (new model + OCR + retrieval quality).

---

## Cross‑cutting refactor (enables 3 & 4)

Extract the generation/agent loop from `MainViewModel` into a reusable **`ChatEngine`**
(model load, KV‑cache session, tool dispatch, streaming). Both the UI and background
workers (Feature 3) and RAG (Feature 4) call it. Do this before/with Feature 3.

## Suggested roadmap
1. **ChatEngine refactor** (small, unblocks later work).
2. **Feature 1 — Share‑to‑assistant** (fast, high daily utility).
3. **Feature 2 — Phone‑native tools** (assistant that acts).
4. **Feature 3 — Scheduled prompts** (proactive value).
5. **Feature 4 — On‑device RAG** (knowledge base; largest).

## Testing philosophy (consistent with the repo)
- Pure logic → host unit tests (like `WorkspaceResolveTest`, `WebToolsTest`).
- Device‑dependent (WebView, whisper, providers) → instrumented tests that are
  deterministic (fixed inputs, assert outputs) — no reliance on flaky UI automation.
- Ship each feature: commit → push → build APK → GitHub release.
